# Rust에서 WIP 코드를 경고로 남기기

- 원문 링크: https://news.hada.io/topic?id=31182
- 발행일: 2026-07-07
- 피드: GeekNews - 개발/기술/스타트업 뉴스 서비스

## 요약

Rust 개발 중 모든 오류 처리와 소유권 문제를 즉시 해결하면 구현 흐름이 끊기기 때문에, happy path를 먼저 완성하고 미완성 부분은 컴파일러 경고로 추적하는 전략을 소개하는 글입니다. 표준 Rust의 `unwrap`, `clone`, `todo!()` 등으로도 미완성 코드를 표시할 수 있지만, 경고를 안정적으로 발생시키지 못하거나 프로덕션 용도와 구분되지 않는 한계가 있습니다. 이를 해결하기 위해 만든 `wip` crate는 `wip!`, `unwrap_wip`, `clone_wip`, `fixme!`를 통해 모든 임시 코드가 항상 경고를 내도록 보장합니다. Meilisearch는 CI에서 `-D warnings`로 경고를 오류로 전환해 WIP 코드의 병합을 차단하고, `wip` 의존성은 리뷰 전 히스토리 정리 단계에서 제거하는 흐름을 사용합니다.

## 핵심 포인트

- **전략의 배경**: 오류 경로를 즉시 정리하면 구현 속도와 집중이 떨어지므로, "나중에 처리할 일"을 경고 수준으로 낮춰 개발을 이어가되 PR 병합 전 정리 신호로 남깁니다.
- **표준 방식의 한계**: `unwrap`과 `clone`은 경고를 전혀 내지 않고, `todo!()`는 간접 원인에 의존하며, `///` 문서 주석 트릭은 위치에 따라 컴파일 오류나 의도치 않은 문서화로 이어집니다.
- **`wip` crate의 도구**: `wip!`(항상 경고를 내는 `todo!` 대체), `unwrap_wip`/`clone_wip`(임시 용도임을 명시하는 extension trait), `fixme!`(panic 없이 경고만 생성하는 메모용 매크로)를 제공하며, deprecation warning 메커니즘을 활용합니다.
- **안전장치**: CI에서 `-D warnings`로 경고를 오류화하면 WIP 코드가 실수로 배포되는 것을 구조적으로 방지할 수 있습니다.
- **커뮤니티 반응**: Clippy 린트(`clippy::todo`, `clippy::unwrap_used`)의 재발명이라는 비판이 있으나, 저자는 `rustc` 경고가 Clippy보다 빠르고 인라인 피드백에 유리하다고 반박했습니다. `fixme!`는 표준 라이브러리에 포함될 가치가 있다는 긍정적 평가와 함께, `todo!()`에 기본 경고가 필요하다는 의견도 제기되었습니다.

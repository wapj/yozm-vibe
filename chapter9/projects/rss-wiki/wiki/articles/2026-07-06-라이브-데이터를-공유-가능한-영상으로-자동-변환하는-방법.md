# 라이브 데이터를 공유 가능한 영상으로 자동 변환하는 방법

- 원문: [https://news.hada.io/topic?id=31167](https://news.hada.io/topic?id=31167)
- 발행일: 2026-07-06
- 피드: GeekNews - 개발/기술/스타트업 뉴스 서비스

## 요약

라이브 데이터가 갱신될 때마다 사람이 영상을 편집하는 대신, 웹 페이지 자체를 영상 원본으로 사용하는 자동화 워크플로를 구축한 사례입니다. 숨겨진 세로형(9:16) 웹 페이지를 Playwright로 제어하며 프레임 단위 스크린샷을 찍고, ffmpeg로 이어 붙여 약 14초 분량의 MP4·GIF를 생성합니다. 실시간 화면 녹화 대신 flipbook 방식으로 프레임을 하나씩 캡처하여 결정적이고 재현 가능한 결과를 얻습니다. 예약된 GitHub Action이 데이터 변경을 감지하면 사이트를 재배포하고 비율·언어별 4개 컷을 CDN에 게시하되, 최종 게시 전에는 사람의 검토 단계를 유지합니다. 전체 시스템은 기존 도구를 연결하는 수백 줄의 glue code로 구성되며 운영 비용은 월 몇 달러 수준입니다.

## 핵심 포인트

- **웹 페이지가 곧 영상 원본**: 별도 편집 프로젝트 없이 사이트 내 숨겨진 전용 페이지(헤드라인, 수치, 애니메이션 차트, 지도)를 활용하고, 다국어 프레임워크도 그대로 상속합니다.
- **단일 playhead 동기화**: `storyboard.ts` 파일 하나가 beat와 지속 시간을 정의하고, 차트·지도·카운터가 하나의 playhead에 묶여 특정 timestamp에서 동일한 상태로 렌더링됩니다.
- **프레임별 "굽기" 방식**: 실시간 녹화는 프레임 누락과 환경별 편차가 발생하므로, 특정 시점의 화면을 스크린샷으로 수백 번 캡처하는 결정적 방식을 채택했습니다.
- **서버 렌더링 안정화**: GPU가 없는 서버에서 MapLibre GL JS를 위해 Chromium 소프트웨어 렌더링을 사용하고, 프레임을 PNG 대신 JPEG로 저장해 속도를 확보합니다.
- **자동화와 사람 검토의 균형**: YouTube는 API로 미공개 초안을 업로드하지만, Instagram은 즉시 공개만 지원하므로 이메일 알림 후 수동 게시하는 승인 단계를 유지합니다.
- **비용 효율**: 데이터 파이프라인, 헤드리스 브라우저, ffmpeg를 연결한 구조로 월 몇 달러에 운영되며, 색상·라벨 변경은 코드 한 줄 수정으로 다음 렌더링에 반영됩니다.

# Gradle이 jj를 쓰지 못한 사소한 이유

- 원문: [https://news.hada.io/topic?id=31213](https://news.hada.io/topic?id=31213)
- 발행일: 2026-07-07
- 피드: GeekNews - 개발/기술/스타트업 뉴스 서비스

## 요약

Gradle 팀은 Git 호환 VCS인 Jujutsu(jj)를 일상 개발에 도입하려 했으나, `jj`가 `.gitattributes`의 파일별 줄바꿈(EOL) 규칙을 읽지 못한다는 사소해 보이는 문제 때문에 채택을 보류했습니다. Windows에서 `gradlew.bat`가 안정적으로 실행되려면 CRLF 줄바꿈이 필요한데, Git은 `*.bat text eol=crlf` 규칙으로 저장소에는 LF, 작업 디렉터리에는 CRLF를 유지하지만 `jj`는 이 차이를 계속 변경 사항(phantom diff)으로 인식합니다. `gradlew.bat`를 CRLF 그대로 커밋하고 `*.bat -text`로 바꾸는 우회책이 있으나, 실수로 LF가 유입되어도 자동 복구되지 않아 Windows 실행 안정성을 보장하기 어렵다고 판단했습니다. Gradle 팀은 당분간 `git worktree`로 다중 작업 디렉터리 요구를 충족하고, `jj`에 `eol` 속성 지원이 구현되면 재검토할 계획입니다.

## 핵심 포인트

- **문제의 근원**: `jj`는 `.gitattributes`를 읽지 않아 `*.bat text eol=crlf` 같은 파일별 EOL 규칙을 적용할 수 없습니다. 전역 설정인 `working-copy.eol-conversion`만 제공되어 `.bat` 파일만 따로 지정할 수 없습니다.
- **phantom diff 발생**: Git은 `gradlew.bat`를 LF로 저장하고 CRLF로 체크아웃하지만, `jj`는 디스크의 CRLF를 LF blob 대비 변경으로 기록하여 가짜 변경이 계속 생깁니다. 한 사용자는 수십만 개 파일에서 이 문제가 발생해 `jj`를 포기했습니다.
- **왜 CRLF가 필수인가**: `cmd.exe`는 LF 전용 배치 파일의 label과 `goto`를 잘못 처리할 수 있어, `gradlew.bat`는 반드시 CRLF로 체크아웃되어야 합니다.
- **우회책의 한계**: `*.bat -text`로 CRLF를 직접 커밋하면 phantom diff는 사라지지만, 편집기가 LF로 바꿔도 Git이 자동 정규화하지 않아 Windows에서 조용히 실행이 실패할 위험이 생깁니다. Gradle 팀은 전체 생태계(초보 개발자의 프로젝트 포함)에 이 전제를 걸기 어렵다고 판단했습니다.
- **대안과 향후 계획**: 다중 작업 디렉터리 요구는 기존의 `git worktree` 패턴으로 충족하고, `jj`의 `eol` 지원(jj-vcs/jj#53, `gix-filter` 기반 구현 논의 중)이 구현되면 다시 시도할 예정입니다.
- **커뮤니티 반응**: Git의 EOL 변환 자체가 설계 실수이며 린터나 CI 검사가 담당할 일이라는 의견과, 반대로 VCS가 이런 변환을 더 적극적으로 해야 한다는 의견이 갈렸습니다. PowerShell 스크립트로의 전환은 실행 정책과 버전 파편화 문제로 현실성이 낮다는 평가가 있었습니다.

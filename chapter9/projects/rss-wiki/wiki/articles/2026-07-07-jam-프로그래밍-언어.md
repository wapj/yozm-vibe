# Jam 프로그래밍 언어

## 요약

Jam은 C 계열 언어의 직관적인 사용감을 유지하면서 GC 없이 안전성과 고성능을 동시에 확보하려는 v1.0 이전 단계의 프로그래밍 언어입니다. 핵심 설계는 mutable value semantics와 Rust식 drop 시스템의 결합으로, 참조나 lifetime 문법을 사용자에게 노출하지 않고 컴파일러가 소유권 추적과 자동 정리를 처리합니다. `undefined`와 암묵적 zero 초기화를 모두 배제하고 `Maybe(T)`와 정적 분석으로 지연 초기화를 다루며, first-class reference가 없어 Jam struct가 C ABI와 자연스럽게 호환되는 점을 강점으로 내세웁니다. 컴파일러는 아직 C++ 부트스트랩 단계로 비공개이며, 108개 프로젝트를 Jam으로 만든 뒤 오픈소스화할 계획입니다. 다만 커뮤니티에서는 정량 데이터 부족, `unsafe` 경계의 허점, 참조 부재로 인한 자료구조 표현력 한계를 지적하는 회의적 반응도 상당합니다.

## 핵심 포인트

- **언어 포지셔닝**: Go/Zig/modern C 수준의 낮은 학습 곡선을 유지하면서 C의 버그 클래스를 언어 차원에서 차단하는 것이 목표입니다. AI 시대에 병목이 코드 작성에서 코드 리뷰로 이동하므로 컴파일러가 더 많은 버그를 잡아야 한다는 문제의식에서 출발했습니다.
- **자동 drop 시스템**: drop을 가진 타입의 binding이 scope를 벗어나면 컴파일러가 drop 호출을 자동 합성합니다. Zig의 `defer`처럼 명시적 정리가 필요 없고, C++ RAII의 복잡성(rule of 0/3/5, 예외 처리 등)도 피합니다. `return`/`break`/`continue` 등 모든 scope 탈출 경로에서 innermost-first 순서로 drop이 실행됩니다.
- **초기화 모델**: 모든 binding은 실제 initializer를 요구하며, 지연 초기화는 `Maybe(T)`의 `empty()`/`write()`/`unsafeAssumeInit()`으로 처리합니다. 분석기가 초기화를 증명하지 못한 `unsafeAssumeInit()` 호출은 컴파일 에러가 됩니다.
- **참조 제거와 parameter mode**: borrow는 함수 호출 프레임 동안만 존재하고 저장·반환·필드 보관이 불가능하므로 lifetime 표기가 불필요합니다. parameter mode는 read-only borrow(기본), `mut`(배타적 borrow), `move`(소유권 이전) 세 가지입니다.
- **C ABI 친화성**: `export` 함수는 C calling convention의 plain 이름으로 노출되고 struct는 C 호환 layout을 가지므로, Rust처럼 `#[repr(C)]`나 unsafe shim layer를 만들 필요가 없다는 점을 내세웁니다.
- **컴파일 속도 설계**: Rust의 다단계 IR 대신 typed IR인 JIR 하나를 사용하고, flat array 기반 자료구조로 cache 친화성을 높였습니다. 다만 벤치마크가 없는 design claim 단계입니다.
- **커뮤니티의 비판**: 정량 데이터 없이 서사로 설득하는 LLM풍 글쓰기에 대한 경계, drop 이후 값 사용 예제의 안전성 허점(`close(7)` 후 `7` 반환), Rust ABI에 대한 오해, "lifetime 없는 Rust" 시도가 반복적으로 실패해 온 역사 등이 지적되었습니다.

## 요약

Claude Code 팀은 코딩 에이전트에 매번 프롬프트를 입력하는 방식에서 벗어나, 정지 조건이 충족될 때까지 작업 사이클을 반복하는 "루프" 운영 패턴으로의 전환을 제안합니다. 루프는 트리거 방식, 정지 방식, 사용하는 primitive, 적합한 작업 유형이라는 네 가지 기준으로 분류되며, Turn-based·Goal-based·Time-based·Proactive 네 유형으로 나뉩니다. 각 유형은 검증, 정지 조건, 트리거, 프롬프트 작성을 단계적으로 에이전트에 위임하는 구조입니다. 다만 모든 작업에 복잡한 루프가 필요한 것은 아니므로 가장 단순한 해법부터 시작해야 하며, 루프의 출력 품질은 코드베이스 청결도와 검증 체계 등 주변 시스템에 좌우되므로 코드 품질 유지와 토큰 사용 관리가 핵심 과제입니다.

## 핵심 포인트

- **루프의 정의**: 정지 조건이 충족될 때까지 작업 사이클을 반복하는 에이전트. 트리거·정지·primitive·적합 작업 네 기준으로 분류합니다.
- **Turn-based 루프**: 사용자 프롬프트로 시작하고 Claude가 완료를 판단하면 정지합니다. 검증 절차를 `SKILL.md`로 인코딩하면 자체 확인 범위를 넓혀 턴 수를 줄일 수 있습니다.
- **Goal-based 루프 (`/goal`)**: 검증 가능한 완료 기준(통과 테스트 수, 점수 임계치 등)을 정의하면 evaluator가 조건을 확인하며 목표 달성 또는 최대 턴 수 도달까지 재작업합니다.
- **Time-based 루프 (`/loop`, `/schedule`)**: 지정 간격으로 프롬프트를 재실행하며, PR 리뷰 대응·CI 실패 수정처럼 외부 시스템과 인터페이스하는 반복 작업에 적합합니다. `/schedule`로 클라우드 routine 전환이 가능합니다.
- **Proactive 루프**: 이벤트나 스케줄로 트리거되어 사람의 실시간 개입 없이 실행됩니다. 버그 분류, 마이그레이션, 의존성 업그레이드 같은 잘 정의된 반복 작업에 적합하며, dynamic workflows·auto mode와 조합해 장기 실행 루프를 구성합니다.
- **코드 품질 유지**: 코드베이스 청결 유지, skills 기반 자체 검증, 문서 접근성 확보, 별도 맥락의 리뷰 에이전트 활용이 권장됩니다. 개별 결과가 기준에 미달하면 개별 수정에 그치지 않고 시스템 자체를 개선해야 합니다.
- **토큰 사용 관리**: 작업 규모에 맞는 primitive와 모델 선택, 명확한 성공·정지 기준 정의, 대규모 실행 전 파일럿, 결정론적 작업의 스크립트 처리, `/usage`·`/workflows`를 통한 사용량 검토가 원칙입니다.
- **시작 방법**: 본인이 병목이 되는 작업 하나를 골라 검증 가능 여부, 목표의 명확성, 스케줄 도착 여부를 기준으로 위임 가능한 부분을 판단하고, 실행 후 정체·과도 실행 지점을 관찰하며 반복 개선합니다.

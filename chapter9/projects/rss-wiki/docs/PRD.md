# RSS Wiki — PRD

## 1. 목적

- 수십 개의 RSS 피드를 자동으로 수집·요약·분류하여 **AI 큐레이팅 매거진**으로 발행한다.
- 매일 점심시간(13:00)에 가볍게 훑어볼 수 있도록 정리한다.
- **주제별 인덱스**로 누적하여 과거 글도 쉽게 다시 찾을 수 있게 한다.

## 2. 사용자 / 사용 환경

- 개인용 도구 (1인 사용).
- 로컬 CLI 실행 환경. 외부 호스팅·계정 시스템·동시 사용자 고려 없음.
- 결과물은 Obsidian / VSCode 등 마크다운 친화 도구로 열람.

## 3. 산출물

- **마크다운 파일** + **git 누적**.
  - 매거진 파일과 인덱스 파일은 모두 마크다운으로만 산출하며, 변경 이력은 git에 남긴다.
- **FastAPI 기반 로컬 웹 인터페이스**(§13)를 함께 제공한다.
  - 매거진/인덱스 열람과 **피드 관리(추가·수정·삭제)** 용도.
  - 마크다운 산출물은 그대로 유지하되, 웹은 SQLite를 직접 조회하여 렌더링한다.

## 4. 매거진 종류와 구성

| 종류     | 발행 시점                   | 구성                                                                           |
| -------- | --------------------------- | ------------------------------------------------------------------------------ |
| **일간** | 매일 13:00                  | 카테고리별 섹션 + 글 1~3줄 요약 카드 + 카테고리별 "오늘의 트렌드 요약" 한 단락 |
| **주간** | 매주 금요일 13:00           | 한 주 동안의 글을 LLM이 다시 통합 요약한 별도 정리글                           |
| **월간** | 매월 마지막 주 금요일 13:00 | 한 달 분량을 LLM이 다시 통합 요약한 별도 정리글                                |

- 주간/월간은 일간을 단순 합치는 것이 아니라, **LLM이 다시 읽고 통합 요약**한다.
- 매거진 하단에는 연속 실패 중인 피드("장애 피드") 목록을 표시한다.

## 5. 동작 흐름 (Daily Pipeline)

1. **12:00** — RSS 수집 시작.
2. 신규 글에 한해 본문 추출 → LLM이 글마다 `(요약, 카테고리, 태그)` 생성.
3. 카테고리별로 묶어 **트렌드 요약 1단락** 생성.
4. **13:00** — 일간 매거진 마크다운 파일 발행.
5. 금요일이면 **주간** 통합 요약 추가 발행, 매월 마지막 주 금요일이면 **월간** 통합 요약도 추가 발행.
6. 주제(카테고리/태그)별 **인덱스 페이지**에 해당 글 카드 append.

### 날짜 기준

- 매거진 그룹핑 및 일간/주간/월간 분류는 **수집일(`fetched_at`)** 을 기준으로 한다.
  - "오늘의 매거진" = 오늘 수집된 글. RSS 항목의 발행일이 며칠 전이어도 오늘 처음 수집됐다면 오늘 매거진에 포함된다.
- RSS 항목의 발행일(`published`/`pubDate`)은 **참고용 메타데이터**로 SQLite에 원본 그대로 저장하고, 매거진 카드에 함께 표기한다(그룹핑 키로는 사용하지 않음).
- 비교/그룹핑 시에는 **로컬 타임존(KST)** 으로 정규화한다.

## 6. 카테고리 / 태그 체계

- **자동 생성 방식**. 사전 고정 카테고리 목록을 두지 않는다.
- 카테고리 결정 시 LLM에게 **DB에 이미 누적된 기존 카테고리 목록을 함께 제공**한다.
  - 가능하면 기존 카테고리에서 선택하도록 유도.
  - 진짜 새로운 주제일 때만 신규 카테고리 생성.
- **태그**는 자유 형식으로 LLM이 부여(검색·탐색용).

## 7. 검색 / 탐색

- 카테고리·태그별 **인덱스 마크다운 페이지**에서 링크로 탐색.
- 본문/요약 검색은 CLI `grep` / `rg` 사용을 전제로 한다.
- **임베딩 기반 의미 검색은 도입하지 않음** (당장은 단순함을 우선).

## 8. 저장소 (Storage)

**SQLite + Markdown 하이브리드.**

- **SQLite** — 운영 데이터의 단일 진실 원천(SoT).
  - 글 메타데이터(원문 URL, 제목, 발행일, 출처 피드 등).
  - 본문/요약/카테고리/태그.
  - 발행 이력(어떤 글이 어느 매거진/인덱스에 들어갔는지).
  - 카테고리·태그 인덱스.
  - 중복 체크 키(정규화된 URL 해시, 제목 해시).
- **Markdown** — 사람이 읽는 산출물.
  - 일간/주간/월간 매거진 파일.
  - 카테고리/태그별 인덱스 페이지.
  - 산출물은 SQLite에서 재생성 가능해야 한다(마크다운은 결과 캐시 성격).

## 9. 수집 / 실패 / 중복 처리

### 수집

- 매일 **12:00**, 1회.
- 피드별 HTTP 타임아웃 적용.

### 실패 정책

- 피드 단위 실패는 **조용히 스킵**하고 로그에 누적.
- **연속 실패 N회** 이상인 피드는 매거진 하단 "장애 피드" 섹션에 노출.
- N의 구체값은 미해결(아래 §12 참고).

### 중복 처리

- **URL 정규화**(쿼리스트링/UTM 등 추적 파라미터 제거) 후 해시 비교.
- **제목 해시** 비교 병행.
- 임베딩 유사도 기반 중복 판정은 도입하지 않는다.

## 10. LLM 통합

- **Claude CLI 헤드리스 모드**(`claude -p "..."`)를 `subprocess`로 호출.
- API 키 관리 불필요(CLI 자체 인증 사용).
- 긴 본문은 가능한 한 **통째로 입력**하고, 임의로 자르지 않는 방향을 우선한다.
- 호출 단위: ① 글당 요약·카테고리·태그 생성, ② 카테고리별 트렌드 요약, ③ 주간·월간 통합 요약.

## 11. 입력 (RSS 피드 목록)

- 피드 목록은 **SQLite의 `feeds` 테이블**을 단일 진실 원천(SoT)으로 한다.
- 초기 부트스트랩용으로 **TOML 시드 파일**(`feeds.toml`)을 두고, 최초 실행 시 DB에 `upsert`한다.
- 운영 중 추가/수정/삭제는 **웹 인터페이스(§13)** 또는 직접 DB 편집으로 처리한다.
- 기본 포함 피드:
  - `https://news.google.com/rss?hl=ko&gl=KR&ceid=KR:ko` (Google News 한국)

### `feeds` 테이블 스키마(요약)

| 컬럼                      | 설명                                         |
| ------------------------- | -------------------------------------------- |
| `id`                      | PK                                           |
| `url`                     | 피드 URL (UNIQUE)                            |
| `title`                   | 표시용 이름 (선택, 미지정 시 피드 메타 사용) |
| `enabled`                 | 수집 대상 여부 (boolean, 기본 true)          |
| `consecutive_failures`    | 연속 실패 카운트                             |
| `last_fetched_at`         | 마지막 수집 시각                             |
| `last_success_at`         | 마지막 성공 시각                             |
| `created_at`/`updated_at` | 생성/수정 시각                               |

## 12. 결정 사항 (2026-05-05 Planner 자체 결정)

PRD 시점 미해결로 남겨둔 항목을 단순함을 우선 기준으로 결정한다.

1. **카테고리 정규화 규칙** — 저장 시 `strip()`과 소문자화만 수행. 한/영 동의어 통합은 별도 규칙 테이블을 두지 않고, LLM에게 기존 카테고리 목록을 함께 제공해 자연어로 흡수되도록 위임한다.
2. **로컬 뷰어 도입 여부** — **FastAPI 기반 로컬 웹 인터페이스 도입**(§13). 마크다운 열람용도 겸하지만, 주된 도입 이유는 피드 관리 UI 제공.
3. **의미 검색 도입 여부** — 도입하지 않는다. `grep`/`rg`로 충분.
4. **본문 추출 실패 시 폴백** — 1차로 RSS `summary` 필드를 사용. `summary`도 없으면 해당 글을 스킵하고 실패 카운트만 누적한다.
5. **연속 실패 임계값 N** — `5`. (일일 수집 기준 약 1주일 연속 실패 시 "장애 피드" 노출.)
6. **RSS 피드 설정 파일 포맷** — `TOML`. Python 3.11+ 표준 라이브러리(`tomllib`)로 외부 의존성 없이 파싱 가능, `pyproject.toml`과 포맷 일관성 유지. 시드 용도로만 사용하고 운영 중 SoT는 SQLite `feeds` 테이블.

## 13. 웹 인터페이스 (FastAPI)

### 목적

- **피드 관리 UI**: 추가/수정/삭제, 활성화 토글, 실패 상태 확인.
- **매거진/인덱스 열람**: 마크다운 산출물을 브라우저에서 바로 렌더링.

### 스택

- **FastAPI** + **Uvicorn**.
- 템플릿: **Jinja2** (서버 사이드 렌더링; SPA 미도입).
- 마크다운 렌더: `markdown-it-py` 또는 `markdown` 패키지.
- 정적 자산: 빌드 단계 없이 운영하되, **모던한 UI**를 위해 단일 CSS 파일로 디자인 시스템을 구성한다.

### UI/UX 요구사항

- **모던한 디자인**으로 개편한다. 기본 정렬·여백·타이포그래피만 있는 수준이 아니라, 실사용에 적합한 시각적 완성도를 갖춘다.
  - 일관된 디자인 토큰(컬러 팔레트, 간격 스케일, 폰트 사이즈, 라운드, 섀도우)을 단일 CSS 파일에서 정의.
  - 라이트/다크 모드를 시스템 설정(`prefers-color-scheme`)에 따라 자동 적용.
  - 카드/리스트/배지/버튼/폼 등 공통 컴포넌트의 스타일을 통일.
  - 본문 영역은 가독성 있는 폭(최대 ~720px)으로 제한하고, 코드/인용/표/링크 스타일을 정리.
  - 반응형(모바일/태블릿/데스크탑) 레이아웃을 지원.
- **상단 GNB(Global Navigation Bar)** 를 모든 페이지에 고정 배치한다.
  - 좌측: 사이트 타이틀(로고 텍스트) — 클릭 시 `/`로 이동.
  - 중앙/우측: 주요 네비게이션 링크 — **매거진**(`/magazines`), **카테고리**(`/categories`), **태그**(태그 인덱스 또는 검색), **피드 관리**(`/feeds`).
  - 현재 활성 메뉴는 시각적으로 강조(active 스타일).
  - 모바일에서는 햄버거 메뉴 또는 가로 스크롤 가능한 탭바로 대체.
- 폼 제출 후 토스트/배너로 처리 결과(성공/실패)를 노출한다(쿼리스트링 플래시 메시지로 구현, JS 없이 동작).

### 라우트

| 메서드 / 경로             | 설명                                                  |
| ------------------------- | ----------------------------------------------------- |
| `GET /`                   | 최신 일간 매거진 + 최근 발행 매거진 목록              |
| `GET /magazines`          | 일간/주간/월간 매거진 인덱스 (발행일 기준 정렬)       |
| `GET /magazines/{slug}`   | 매거진 단건 렌더 (마크다운 → HTML)                    |
| `GET /categories`         | 카테고리 인덱스                                       |
| `GET /categories/{name}`  | 해당 카테고리 글 목록 (인덱스 마크다운 렌더)          |
| `GET /tags/{name}`        | 해당 태그 글 목록                                     |
| `GET /feeds`              | 피드 목록 페이지 (관리 UI, 인라인 추가 폼 포함)       |
| `GET /feeds/new`          | 피드 추가 폼 (별도 페이지)                            |
| `POST /feeds`             | 피드 추가 (form: `url`, `title`, `enabled`)           |
| `GET /feeds/{id}/edit`    | 피드 수정 폼 (`url`/`title`/`enabled` 모두 편집 가능) |
| `POST /feeds/{id}`        | 피드 수정 적용                                        |
| `POST /feeds/{id}/delete` | 피드 삭제                                             |
| `POST /feeds/{id}/toggle` | 활성/비활성 토글                                      |
| `POST /feeds/{id}/reset`  | 연속 실패 카운트 리셋                                 |
| `GET /healthz`            | 헬스체크                                              |

- 폼 제출은 모두 `application/x-www-form-urlencoded` 기반 (HTMX/JS 없이 동작).
- 모든 변경 라우트는 처리 후 `303 See Other`로 목록 페이지로 리다이렉트.

### 피드 CRUD 동작

피드 관리 페이지(`/feeds`)에서 **추가·수정·삭제** 모두 웹 UI로 가능해야 한다(기존 CLI/DB 직접 편집 의존 제거).

- **추가**: `/feeds` 상단의 추가 폼 또는 별도 페이지(`GET /feeds/new`). `url`은 필수, `title`/`enabled`는 선택. URL 정규화 후 `feeds.url` UNIQUE 제약으로 중복 방지. 즉시 다음 수집 사이클부터 반영.
- **수정**: 목록의 각 행에 "수정" 버튼 → `GET /feeds/{id}/edit` 폼. **`url`도 수정 가능**하도록 한다(변경 시 정규화 후 UNIQUE 검증, 충돌 시 폼에 에러 표시). `title`, `enabled`도 함께 수정.
- **삭제**: 목록의 각 행에 "삭제" 버튼(확인 다이얼로그 또는 별도 확인 단계). 소프트 삭제 없이 **하드 삭제**. 단, 이미 수집·발행된 글은 그대로 유지(글의 `feed_id`는 nullable로 두거나, 삭제 시점에 피드 메타를 글 row에 스냅샷).
- **활성화 토글**: 목록에서 토글 버튼/스위치로 즉시 변경. `enabled=false`인 피드는 수집 사이클에서 스킵. 카운터는 유지.
- **실패 카운트 리셋**: 연속 실패 카운트가 0보다 큰 피드 행에 "리셋" 버튼 노출.

피드 목록 화면에는 다음 정보를 한눈에 표시한다: `title`, `url`, `enabled` 상태, `consecutive_failures`, `last_fetched_at`, `last_success_at`, 그리고 행별 액션 버튼(수정/삭제/토글/리셋).

### 인증 / 접근 범위

- 개인용·로컬 전용. 기본 바인딩은 **`127.0.0.1`** 로만 listen, 인증 없음.
- 외부 노출 시나리오는 PRD 범위 외.

### 실행

- `uvicorn rss_wiki.web:app --host 127.0.0.1 --port 8765` 또는 동등 CLI 서브커맨드(`rss-wiki web`).
- 수집/발행 파이프라인과는 **별도 프로세스**. 동일 SQLite 파일을 공유하며 쓰기 충돌 방지를 위해 WAL 모드 사용.

선택: A. Canvas 2D API 직접 사용

## 근거

PRD의 목표에 비추어 독립적으로 판단한다.

- 규모: 출전마 4~8마리와 소량의 파티클(오라·스피드라인·폭죽)은 Canvas 2D
  API의 즉시 모드 드로잉으로 60fps를 확보하기에 충분한 부하다. WebGL 가속
  (Pixi.js)이 주는 렌더링 처리량 이점을 필요로 하는 규모가 아니다.
- 의존성·번들: Pixi.js는 수백 KB 규모의 런타임 의존성을 추가한다. Canvas 2D
  직접 구현은 외부 렌더링 라이브러리를 두지 않아 번들 크기와 공급망 표면을
  최소화한다.
- 배포: 두 방식 모두 정적 호스팅이 가능하나, 의존성이 없는 쪽이 빌드·배포
  구성을 단순하게 유지한다.
- 표현력: 오라(radial gradient), 스피드라인(선분), 폭죽(원·투명도 감쇠)은
  모두 Canvas 2D의 기본 API로 구현 가능하다.

WebGL이 주는 대량 파티클·셰이더 이점이 이 프로젝트의 요구를 초과하므로,
표현력을 희생하지 않으면서 의존성을 최소화하는 A를 선택한다.
